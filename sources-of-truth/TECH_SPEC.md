# TECH_SPEC — Техническая спецификация DocParser

## Общая техническая концепция

Локальное веб-приложение с раздельным фронтендом и бэкендом, запускаемое через Docker Compose.

Ключевые принципы:
- простота и минимализм — MVP без overengineering;
- конфигурация через переменные окружения;
- устойчивость к изменениям формы входных документов;
- разделение логики парсинга, маппинга и генерации.

---

## Архитектурный стиль

**Monolith (два контейнера: frontend + backend)**

Коммуникация:
- Sync: HTTP/REST (frontend → backend API)
- Async: не требуется для MVP

## API-контракт

### POST /api/process
- Принимает: `multipart/form-data` с DOCX-файлами (входные документы + шаблоны).
- Возвращает: ZIP-архив с заполненными документами + JSON-отчёт.
- Ошибки: 400 если не DOCX, 422 если не удалось распарсить, 500 при внутренней ошибке.

### GET /api/health
- Возвращает: `{"status": "ok"}`

## Подход к парсингу и генерации

### Парсинг (извлечение данных из входных документов)
1. **Семантический поиск полей** — ищем по ключевым словам/фразам в тексте и таблицах, а не по индексам строк.
2. **Fallback** — если поле не найдено семантически, пробуем по позиции в таблице (как дополнительный механизм, не единственный).

### Маппинг (единая конфигурация `FIELD_MAP`)
3. **Единый конфигурируемый маппинг** — `mapping/field_map.py` содержит список записей, каждая связывает:
   - плейсхолдер в шаблоне (например `[ИНН]`)
   - источник данных (`supplier_card`, `customer_request`, `commercial_terms`)
   - поле DTO (например `inn`)
   - трансформацию (опционально: `format_money`, `shorten_name` и т.д.)
4. Это **единственное место**, где описано соответствие плейсхолдеров и данных.
5. Добавление нового поля = добавление строки в `FIELD_MAP`, без правки кода генератора или парсеров.

### Генерация (заполнение шаблонов)
6. Генератор **сканирует шаблон** — находит все плейсхолдеры вида `[...]`.
7. Каждый найденный плейсхолдер **ищется в FIELD_MAP** — по точному совпадению.
8. Значение **извлекается из DTO** по `source` + `field`, применяется трансформация.
9. Если плейсхолдер не найден в маппинге — попадает в `unmapped` в отчёте.

### Отчёт
10. Каждое поле логируется: `found` (заполнено, с источником) / `warnings` (данные не найдены) / `unmapped` (плейсхолдер без маппинга).

## Ограничения (важно для агента)

- ❌ не добавлять технологии без явной причины и согласования;
- ❌ не использовать внешние LLM API;
- ❌ не писать бизнес-логику во frontend;
- ❌ не хардкодить индексы строк/ячеек как единственный механизм парсинга;
- ❌ не хардкодить плейсхолдеры шаблонов в коде генератора — только через `FIELD_MAP`;
- ✅ придерживаться rule-based подхода к парсингу;
- ✅ код минимальный, чёткий, явный;
- ✅ конфигурация вынесена наружу (env-переменные, маппинг полей).
