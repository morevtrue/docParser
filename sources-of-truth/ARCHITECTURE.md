# ARCHITECTURE — Архитектура DocParser

## Назначение документа

Описывает:
- какие компоненты существуют и за что отвечают;
- как компоненты взаимодействуют;
- какие правила нельзя нарушать.

---

## Архитектурные принципы

### Разделение ответственности
- Frontend — только UI: загрузка файлов, отображение результата, скачивание.
- Backend — вся логика: парсинг, маппинг, генерация, отчёт.
- Парсинг, маппинг и генерация — отдельные модули внутри бэкенда.

### Устойчивость к изменениям форм
- Парсинг основан на семантическом поиске (ключевые слова), а не на жёстких индексах.
- Маппинг полей вынесен в конфигурацию — можно менять без правки кода парсеров.

### Честность данных
- Система не выдумывает значения.
- Каждое поле в отчёте: найдено (с источником) или не найдено (warning).

---

## Общая схема

```
[ Браузер (React + shadcn) ]
          |
    POST /api/process (multipart/form-data)
          |
[ Backend (FastAPI) ]
    |         |           |
[ Parser ] [ Mapper ] [ Generator ]
    |         |           |
    ↓         ↓           ↓
Извлечение → Pydantic  → Заполнение
данных из    DTO          шаблонов
DOCX                      → ZIP + отчёт
```

## Компоненты

### Frontend (React + shadcn)
- Загрузка DOCX-файлов (drag & drop или file picker).
- Валидация формата на клиенте (только .docx).
- Отправка файлов на бэкенд.
- Отображение отчёта об обработке.
- Скачивание ZIP с результатами.
- Проксирование `/api/*` на бэкенд через Vite dev proxy.
- **Не содержит бизнес-логики.**

### Backend (FastAPI)
- Единственная точка входа: `POST /api/process`.
- Health-check: `GET /api/health`.
- Оркестрирует pipeline: парсинг → маппинг → генерация → упаковка.

### Parser (модуль парсинга)
- Отдельный парсер для каждого типа входного документа.
- Извлекает данные из параграфов и таблиц DOCX.
- Использует семантический поиск по ключевым словам.
- Возвращает сырые извлечённые данные + лог парсинга.

### Mapper (модуль маппинга)
- Единая конфигурация `FIELD_MAP` в `mapping/field_map.py`.
- Каждая запись связывает: плейсхолдер шаблона → источник данных → поле DTO → трансформация.
- Единственное место, где описано соответствие плейсхолдеров и данных.
- Добавление нового поля = одна строка в конфиге, без правки кода.

### Generator (модуль генерации)
- Сканирует шаблон DOCX — находит все плейсхолдеры вида `[...]`.
- Ищет каждый плейсхолдер в `FIELD_MAP`.
- Извлекает значение из Pydantic DTO, применяет трансформацию.
- Плейсхолдеры без маппинга → `unmapped` в отчёте.
- Формирует ZIP с результатами + JSON-отчёт.
- **Не содержит хардкода плейсхолдеров — вся логика "что куда" в FIELD_MAP.**

## Границы ответственности

### ❌ Запрещено:
- Frontend содержит логику парсинга или генерации.
- Парсеры зависят от конкретных индексов строк как единственного механизма.
- Генератор содержит хардкод плейсхолдеров — вся логика "что куда" только в `FIELD_MAP`.
- Система выдумывает данные для пустых полей.
- Использование внешних LLM API.

### ✅ Разрешено:
- Добавление новых парсеров для новых типов документов.
- Расширение маппинга полей через конфигурацию.
- Добавление новых шаблонов без изменения кода генератора.
