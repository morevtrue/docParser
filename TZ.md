**DocParser** v1.0 (2025-04-10)
Локальная система автозаполнения тендерных документов по входящим документам

**Контекст**
- Нужно реализовать локальное решение для автозаполнения тендерных документов.
- Пользовательский сценарий должен быть максимально близок к реальной работе: пользователь
загружает документы, а система формирует готовые документы на выходе.

**Важно**: конечный пользователь не должен вручную готовить или загружать JSON как входной
формат. Если вам нужен промежуточный структурированный слой данных, используйте его
только внутри своего решения.

**Что есть**
- Основной набор документов от заказчика и поставщика. (папка "Основной набор")
- Контрольный набор документов с той же фактурой, но с немного изменённой формой и
перестановкой полей. (папка "Контрольный набор с измененной формой")
- 3 пустых шаблона целевых документов. (папка "Шаблоны")
- 3 заполненных эталонных документа для сверки результата. (папка "Заполненные эталоны")

**Основная задача**
- Нужно собрать локальное решение, которое по входящим документам автоматически
формирует 3 документа: Анкета участника, Заявка на участие в закупке, Предложение о цене
договора.
- На входе пользователь загружает только документы. На выходе получает заполненные
документы в формате DOCX.
- Решение должно отрабатывать на основном наборе и тем же способом на контрольном наборе с
изменённой формой.

**Структура входных/выходных данных**
- Вход: Запрос ТКП от заказчика + Карточка поставщика + Коммерческие условия
- Шаблоны: пользователь загружает шаблоны целевых документов вместе с остальными документами.
- Выход: 3 заполненных документа по шаблонам + отчёт об обработке (см. "Отчёт об обработке")

**Обязательные требования**
- Проект должен запускаться локально. Используем Docker-контейнер.
- Нельзя выдумывать данные: если поле не найдено, оно должно остаться пустым, null или
попадать в warning/report.
- Проект должен быть устойчив к небольшим изменениям форм и перестановке полей в
исходных документах.
- **Нельзя** строить проект только на жёстких индексах строк, абзацев или ячеек как единственном
механизме.
- Логика заполнения должна быть отделена от основного кода настолько, насколько это разумно в
рамках MVP.
- Если загружен не DOCX — выдать ошибку "Неправильный формат документа (Требуется DOCX)" в модальном окне с кнопкой "Загрузить другой документ".

**Отчёт об обработке**
- После генерации документов система формирует отчёт, отображаемый в UI:
  - Список успешно заполненных полей.
  - Список полей, для которых данные не найдены (warnings).
  - Ошибки парсинга, если были.
- Отчёт также доступен для скачивания в формате JSON.

**API-контракт (базовый)**
- `POST /api/process` — принимает `multipart/form-data` с входными DOCX-файлами.
  Возвращает ZIP-архив с заполненными документами + JSON-отчёт об обработке.
- `GET /api/health` — healthcheck для Docker.

**Критерии готовности**
- Понятный запуск: локальное web-приложение с использованием Docker.
- Есть объяснимая логика извлечения данных и заполнения документов.
- Есть обработка отсутствующих данных и явные ограничения решения.
- Есть путь к масштабированию на новые шаблоны и новые входящие формы.
- Есть README.md с инструкцией запуска и проверки.
- Визуальная демонстрация: 5–8 скриншотов и/или короткая запись экрана.

**Что используем**
- Стек: фронтенд: React + shadcn (документация: https://ui.shadcn.com/docs/installation).
  Backend: Python + FastAPI (python-docx (парсинг/генерация DOCX), Pydantic (DTO-слой)).
  Deploy: Docker Compose (отдельные контейнеры для фронтенда и бэкенда).
- Внутренний промежуточный JSON/DTO/структуру данных — можно использовать, но только как внутренний
слой.
- Детерминированный подход, rule-based parsing, XML/DOCX processing, semantic matching,
локальные модели — допустимо. **Нежелательно** строить решение как «попросить внешнюю LLM всё заполнить целиком». Если вы
всё же используете AI-слой, опишите это отдельно и обеспечьте воспроизводимый локальный/безопасный сценарий демо.

**Что должно быть по итогу:**
- README с инструкцией запуска.
- Сгенерированные итоговые документы по основному набору.
- Сгенерированные итоговые документы или отдельный отчёт/демо по контрольному набору.
- Краткую доку по архитектуре и ограничениям.

**Целевая логика решения**

Пользовательский сценарий:
1. Пользователь загружает только документы: документы от заказчика, документы поставщика, шаблоны целевых документов.
2. Пользователь **не** подготавливает JSON вручную.
3. Система сама извлекает нужные данные, собирает внутреннюю структуру данных и формирует выходные документы.
4. Если часть данных не найдена — поле остаётся пустым, `null` или попадает в warning/report.
5. Поля в источниках могут меняться местами, формы — незначительно меняться по структуре. Решение должно быть устойчиво к таким изменениям.

Что нужно показать в сдаче:
- Работа на **основном наборе**.
- Работа тем же решением на **контрольном наборе с изменённой формой** без переписывания core-логики.
- Итоговые DOCX.
- Визуальная демонстрация: 5–8 скриншотов и/или короткая запись экрана.

**Допущения**
- В рамках тестового задания не требуется покрыть все возможные DOCX-edge-cases мира.
- Важно показать архитектурно правильный подход, который можно развивать дальше без
переписывания с нуля.
- Допустимо использовать внутренний промежуточный слой данных (JSON/DTO), но он не должен быть пользовательским входом.
