# AI Agent Rules — DocParser

Ты работаешь над проектом DocParser — локальная система автозаполнения тендерных документов.

Ты ОБЯЗАН строго следовать документам ниже:

ИСТОЧНИКИ ИСТИНЫ (SOURCE OF TRUTH):
- sources-of-truth/BUSINESS_SPEC.md
- sources-of-truth/TECH_SPEC.md
- sources-of-truth/ARCHITECTURE.md
- TZ.md

Эти документы имеют приоритет над любыми твоими предположениями,
паттернами или "лучшими практиками".

----------------------------------------------------------------

НАВИГАЦИЯ АГЕНТА:
- Документы по умолчанию:
  - agent/START.md
  - agent/*
  - phases/*
- Остальные документы (включая sources-of-truth/*) — только по запросу,
  когда реально нужны для конкретной задачи.

----------------------------------------------------------------

ЯЗЫК:
- Общение: русский
- Комментарии в коде: русский
- Рассуждения / thinking: русский

----------------------------------------------------------------

ОБЩИЕ ПРАВИЛА:
- НЕ придумывай новые фичи, сущности или сервисы.
- НЕ упрощай и не переинтерпретируй требования.
- НЕ генерируй код, который явно не запрошен.
- НЕ создавай новые файлы без явного указания.
- НЕ переименовывай существующие файлы и папки.
- НЕ предлагай альтернативные архитектуры или стеки.
- НЕ используй внешние LLM API.
- Если что-то непонятно или отсутствует — СПРОСИ, прежде чем действовать.

----------------------------------------------------------------

ПРОДУКТОВЫЕ ПРАВИЛА:
- Всегда думай о том, как реальный пользователь (специалист по тендерам) использует систему.
- Пользователь загружает DOCX → получает заполненные DOCX. Всё.
- Система НЕ выдумывает данные. Ненайденное поле = пустое + warning.

----------------------------------------------------------------

ТЕХНИЧЕСКИЕ ПРАВИЛА:
- Стек: React + shadcn (фронт), Python + FastAPI (бэк), Docker Compose.
- Парсинг: семантический поиск по ключевым словам, НЕ только жёсткие индексы.
- Маппинг полей: конфигурируемый, отделён от кода парсеров.
- Бизнес-логика только на бэкенде. Frontend — только UI.
- Конфигурация через env-переменные.

----------------------------------------------------------------

ПРАВИЛА КОДА:
- Пиши минимальный, чёткий и явный код.
- Предпочитай простые реализации абстракциям.
- Никакого overengineering.
- Строго следуй существующей структуре проекта.
- Health-check эндпоинт обязателен.

----------------------------------------------------------------

ПРОЦЕССНЫЕ ПРАВИЛА:
Перед написанием любого кода:
1. Перечитай релевантные разделы спецификаций.
2. Кратко объясни план, если задача нетривиальная.
3. Реализуй ТОЛЬКО то, что явно запрошено.

Если задача нарушает любое из правил выше — СТОП, попроси уточнения.

Ты — исполнитель, а не принимающий решения.
