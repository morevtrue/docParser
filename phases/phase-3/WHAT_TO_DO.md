# Phase 3 — Генерация выходных DOCX

**Статус:** ✅ Завершена

---

## Цель фазы

Реализовать заполнение шаблонов DOCX извлечёнными данными. После этой фазы
`POST /api/process` принимает входные документы + шаблоны и возвращает ZIP
с заполненными документами + JSON-отчёт.

**Ключевой принцип:** система сама читает плейсхолдеры из шаблонов и ищет
значения через конфигурируемый маппинг. Никакого хардкода плейсхолдеров в коде.

---

## Задачи

### 1. Единый конфигурируемый маппинг (mapping/field_map.py)

Объединить маппинг в одну конфигурацию, которая связывает три вещи:
- **плейсхолдер в шаблоне** (например `[Полное наименование участника]`)
- **источник данных** (например `supplier_card`)
- **поле DTO** (например `full_name`)
- **трансформация** (опционально: `shorten_name`, `format_money`, `join_lines` и т.д.)

Пример структуры:
```python
FIELD_MAP = [
    {
        "placeholder": "[Полное наименование участника]",
        "source": "supplier_card",
        "field": "full_name",
    },
    {
        "placeholder": "[ИНН]",
        "source": "supplier_card",
        "field": "inn",
    },
    {
        "placeholder": "[Итого без НДС]",
        "source": "commercial_terms",
        "field": "total_without_vat",
        "transform": "format_money",
    },
    {
        "placeholder": "[ФИО подписанта]",
        "source": "supplier_card",
        "field": "signatory",
        "transform": "shorten_name",
    },
    # ... все остальные поля
]
```

Этот маппинг — единственное место, где описано соответствие плейсхолдеров и данных.
Добавление нового поля = добавление строки в конфиг, без правки кода генератора.

### 2. Динамический генератор (generator/filler.py)

Переписать `filler.py` так, чтобы он:

1. **Сканировал шаблон** — находил все плейсхолдеры вида `[...]` в параграфах и таблицах.
2. **Искал каждый плейсхолдер в FIELD_MAP** — по точному совпадению строки.
3. **Извлекал значение из DTO** — по `source` + `field` из маппинга.
4. **Применял трансформацию** — если указана (`format_money`, `shorten_name` и т.д.).
5. **Подставлял значение** — или оставлял пустым + warning, если данных нет.
6. **Логировал** — какие плейсхолдеры найдены в шаблоне, какие заполнены, какие нет.

Убрать `_build_replacements()` — больше не нужен. Вся логика "что куда" — в FIELD_MAP.

Для таблицы номенклатуры (Предложение о цене) — оставить специальную логику
`fill_price_template()`, но тоже через конфиг: описать маппинг колонок таблицы
(колонка 1 → item.name, колонка 3 → item.nmc и т.д.).

### 3. Упаковка результатов

Собрать ZIP-архив:
- 3 заполненных DOCX
- report.json — отчёт об обработке

Обновить `POST /api/process`:
- Полный pipeline: парсинг → маппинг → генерация → ZIP
- Content-Type: application/zip
- Отчёт включает:
  - `found` — заполненные поля (плейсхолдер, значение, источник)
  - `warnings` — плейсхолдеры из шаблона, для которых нет данных
  - `unmapped` — плейсхолдеры из шаблона, которых нет в FIELD_MAP (новые/неизвестные)
  - `errors` — ошибки парсинга/генерации

### 4. Сверка с эталонами

Проверить результат на основном наборе:
- Сравнить заполненные документы с эталонами из `Заполненные эталоны/`
- Зафиксировать расхождения (если есть)

## Что должно измениться по сравнению с текущей реализацией

| Было (хардкод) | Стало (динамический маппинг) |
|---|---|
| `_build_replacements()` — жёсткий словарь плейсхолдер → значение | Убрать. Маппинг берётся из `FIELD_MAP` |
| Плейсхолдеры захардкожены в `filler.py` | Плейсхолдеры читаются из шаблона, ищутся в `FIELD_MAP` |
| Добавление поля = правка кода в двух местах | Добавление поля = одна строка в `FIELD_MAP` |
| `field_map.py` не связан с генератором | `field_map.py` — единственный источник маппинга для генератора |

---

## Ограничения

- НЕ реализуем UI — тестируем через curl / Swagger.
- НЕ проверяем контрольный набор — это Phase 5.
- Сохранение форматирования шаблона — best effort (не все edge cases).
- Многострочные плейсхолдеры (банковские реквизиты) — допустимо обрабатывать как составной плейсхолдер через трансформацию `join_lines`.
