# DocParser — Локальная система автозаполнения тендерных документов

## Описание

DocParser принимает входные документы (от заказчика и поставщика) и шаблоны целевых документов,
извлекает данные и формирует заполненные тендерные документы в формате DOCX.

Система работает на rule-based подходе с семантическим поиском полей — устойчива к изменениям
структуры и порядка полей во входных документах.

## Стек

- Frontend: React + shadcn/ui + Vite
- Backend: Python + FastAPI + python-docx + Pydantic
- Deploy: Docker Compose

## Запуск

```bash
docker compose up -d --build
```

- Фронтенд: http://localhost:5173
- API docs (Swagger): http://localhost:8000/docs

## Как проверить

1. Откройте http://localhost:5173
2. Загрузите документы в три зоны:
   - **Документы от заказчика** — Запрос ТКП (DOCX)
   - **Документы поставщика** — Карточка поставщика + Коммерческие условия (DOCX)
   - **Шаблоны** — пустые шаблоны для заполнения (DOCX)
3. Нажмите **Обработать**
4. Просмотрите отчёт (заполненные поля / не найденные)
5. Нажмите **Скачать результат** — получите ZIP с заполненными DOCX

Тестовые документы находятся в папках:
- `Основной набор/` — основной набор документов
- `Контрольный набор с измененной формой/` — контрольный набор с изменённой структурой
- `Шаблоны/` — шаблоны для заполнения
- `Заполненные эталоны/` — эталоны для сверки результата

## Архитектура pipeline

```
[ Браузер (React + shadcn) ]
          |
    POST /api/process (multipart/form-data)
          |
[ Backend (FastAPI) ]
    |         |           |
[ Parser ] [ Mapper ] [ Generator ]
    |         |           |
    ↓         ↓           ↓
Извлечение → Pydantic  → Заполнение
данных из    DTO          шаблонов
DOCX                      → ZIP + отчёт
```

**Шаг 1 — Классификация:** каждый загруженный файл классифицируется по ключевым словам
в первых параграфах (запрос ТКП / карточка поставщика / коммерческие условия / шаблон).

**Шаг 2 — Парсинг:** каждый тип документа парсится своим парсером.
Используется семантический поиск по ключевым словам — не жёсткие индексы строк.
Поддерживаются объединённые поля (ИНН/КПП/ОГРН в одной ячейке и т.п.).

**Шаг 3 — Маппинг:** единая конфигурация `FIELD_MAP` в `mapping/field_map.py`
связывает плейсхолдеры шаблонов с полями DTO.

**Шаг 4 — Генерация:** генератор сканирует шаблон, находит плейсхолдеры `[...]`,
ищет каждый в `FIELD_MAP`, подставляет значение из DTO.

**Шаг 5 — Отчёт:** каждое поле логируется: `found` / `warnings` / `unmapped`.

## Как добавить новый тип документа

1. Создать парсер в `backend/app/parser/new_type.py` по образцу существующих.
2. Добавить поля в `ExtractedData` в `models/schemas.py`.
3. Добавить ключевые слова классификатора в `CLASSIFIERS` в `main.py`.
4. Добавить записи в `FIELD_MAP` в `mapping/field_map.py`.

## Как расширить маппинг полей

Добавить строку в `FIELD_MAP` в `backend/app/mapping/field_map.py`:

```python
{"placeholder": "[Новое поле]", "source": "supplier_card", "field": "new_field"},
```

Код генератора и парсеров трогать не нужно.

## API

### POST /api/process
Принимает `multipart/form-data` с DOCX-файлами.
Возвращает ZIP-архив с заполненными документами и `report.json`.

- `400` — файл не является DOCX
- `500` — внутренняя ошибка

### GET /api/health
Возвращает `{"status": "ok"}`.

## Известные ограничения

- Аутентификация не реализована (MVP, одна роль).
- История обработок не сохраняется (stateless).
- Предпросмотр документов не реализован.
- Классификация документов основана на ключевых словах в первых параграфах —
  документы без стандартных заголовков могут не распознаться.
- Если шаблон содержит плейсхолдер, разбитый на несколько runs в DOCX,
  замена может не сработать (ограничение python-docx).
